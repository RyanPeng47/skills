# 流程图模板库

本文档提供产品设计中常用的 Mermaid 流程图模板，用于可视化业务逻辑和数据流。

## 适用场景速查

| 图类型 | 适用场景 | 复杂度 |
|--------|----------|--------|
| 用户旅程图 | 展示用户操作路径、决策分支 | 低 |
| 序列图 | 展示系统间交互、API调用流程 | 中 |
| 状态机图 | 展示实体状态流转（订单、审批等） | 中 |
| 数据流图 | 展示数据在系统中的流动 | 高 |

---

## 1. 用户旅程流程图 (flowchart)

**用途**: 展示用户从入口到完成目标的完整路径，包含决策分支。

### 模板

```mermaid
flowchart TD
    A[用户入口] --> B{条件判断}
    B -->|条件1| C[操作A]
    B -->|条件2| D[操作B]
    C --> E[结果页]
    D --> E
    E --> F{是否继续?}
    F -->|是| A
    F -->|否| G[结束]
```

### 示例：登录注册流程

```mermaid
flowchart TD
    A[访问应用] --> B{已登录?}
    B -->|是| C[进入主页]
    B -->|否| D[登录/注册页]
    D --> E{选择方式}
    E -->|手机号| F[输入手机号]
    E -->|第三方| G[OAuth授权]
    F --> H[验证码验证]
    H --> I{验证成功?}
    I -->|是| C
    I -->|否| J[错误提示] --> F
    G --> K{授权成功?}
    K -->|是| C
    K -->|否| D
```

## 2. 系统交互序列图 (sequenceDiagram)

**用途**: 展示前后端、微服务之间的调用顺序和数据流向。

### 模板

```mermaid
sequenceDiagram
    participant U as 用户
    participant F as 前端
    participant B as 后端
    participant D as 数据库

    U->>F: 用户操作
    F->>B: API请求
    B->>D: 数据查询
    D-->>B: 返回数据
    B-->>F: 响应结果
    F-->>U: 界面更新
```

### 示例：订单创建流程

```mermaid
sequenceDiagram
    participant U as 用户
    participant F as 前端
    participant O as 订单服务
    participant I as 库存服务
    participant P as 支付服务
    participant D as 数据库

    U->>F: 点击下单
    F->>O: POST /orders
    O->>I: 检查库存
    I->>D: 查询库存
    D-->>I: 库存数量

    alt 库存充足
        I-->>O: 库存OK
        O->>D: 创建订单(待支付)
        D-->>O: 订单ID
        O->>P: 创建支付单
        P-->>O: 支付链接
        O-->>F: 返回支付信息
        F-->>U: 跳转支付
    else 库存不足
        I-->>O: 库存不足
        O-->>F: 400 库存不足
        F-->>U: 提示库存不足
    end
```

## 3. 状态机图 (stateDiagram-v2)

**用途**: 展示业务实体的状态流转，适用于订单、审批、工单等场景。

### 模板

```mermaid
stateDiagram-v2
    [*] --> 初始状态
    初始状态 --> 中间状态: 触发事件
    中间状态 --> 最终状态: 触发事件
    中间状态 --> 异常状态: 异常事件
    最终状态 --> [*]
    异常状态 --> [*]
```

### 示例：订单状态流转

```mermaid
stateDiagram-v2
    [*] --> 待支付: 创建订单
    待支付 --> 已支付: 支付成功
    待支付 --> 已取消: 超时/用户取消
    已支付 --> 待发货: 商家确认
    待发货 --> 已发货: 发货
    已发货 --> 已签收: 物流签收
    已签收 --> 已完成: 确认收货
    已签收 --> 退款中: 申请退款
    退款中 --> 已退款: 退款成功
    已完成 --> [*]
    已取消 --> [*]
    已退款 --> [*]
```

### 示例：审批流程状态

```mermaid
stateDiagram-v2
    [*] --> 草稿
    草稿 --> 待审批: 提交
    待审批 --> 审批中: 开始审批
    审批中 --> 已通过: 通过
    审批中 --> 已驳回: 驳回
    已驳回 --> 草稿: 重新编辑
    已通过 --> [*]
    草稿 --> 已撤销: 撤销
    待审批 --> 已撤销: 撤销
    已撤销 --> [*]
```

## 4. 数据流图 (flowchart)

**用途**: 展示数据在系统各模块间的流动方向。

### 示例：用户数据流

```mermaid
flowchart LR
    subgraph 数据采集
        A[用户行为] --> B[埋点SDK]
        B --> C[日志收集]
    end

    subgraph 数据处理
        C --> D[实时流处理]
        C --> E[批量ETL]
        D --> F[实时数仓]
        E --> G[离线数仓]
    end

    subgraph 数据应用
        F --> H[实时大屏]
        G --> I[BI报表]
        G --> J[用户画像]
    end
```

## 5. 架构概览图 (flowchart)

**用途**: 展示系统整体架构和模块关系。

### 示例：典型Web应用架构

```mermaid
flowchart TB
    subgraph 客户端
        A[Web浏览器]
        B[移动App]
    end

    subgraph 接入层
        C[CDN]
        D[负载均衡]
        E[API网关]
    end

    subgraph 服务层
        F[用户服务]
        G[订单服务]
        H[支付服务]
    end

    subgraph 数据层
        I[(MySQL)]
        J[(Redis)]
        K[(ES)]
    end

    A --> C --> D --> E
    B --> D
    E --> F & G & H
    F & G & H --> I & J
    G --> K
```

## 使用建议

1. **MVP阶段**: 优先画用户旅程图 + 1个核心业务序列图
2. **有状态实体**: 必须画状态机图，明确所有状态和转换条件
3. **复杂系统**: 补充数据流图和架构图
4. **图的粒度**: 一张图聚焦一个主题，避免过于复杂

## 命名规范

- 文件名: `flow-{功能名}.md` 或直接嵌入PRD
- 图标题: 使用中文，简洁明了
- 节点命名: 动词+名词，如"创建订单"、"验证身份"
