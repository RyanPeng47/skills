# Agent 开发阶段依赖关系图

```mermaid
graph TD
    %% 定义阶段节点
    S1[阶段1: 需求背景] --> S2[阶段2: 现有流程梳理]
    S2 --> S3[阶段3: 用户旅程 & Agent Workflow]
    
    %% 并行分支：设计与技术
    S3 --> S4[阶段4: 原型设计]
    S3 --> S5[阶段5: 功能框架]
    
    subgraph 设计流
        S4 --> S6[阶段6: 详细功能点]
    end
    
    subgraph 技术流
        S5 --> S7[阶段7: 技术架构设计]
    end
    
    %% 汇聚点：核心Agent实现
    S6 --> S8[阶段8: 提示词设计]
    S7 --> S8
    
    %% 迭代闭环
    S8 --> S9[阶段9: 评测体系]
    S9 -.->|评测不达标| S8
    S9 -.->|架构瓶颈| S7
    
    %% 保障与上线
    S9 --> S10[阶段10: 非功能需求]
    S10 --> S11[阶段11: 项目计划]
    S11 --> S12[阶段12: 上线运维与持续优化]

    %% 线上反馈闭环
    S12 -.->|线上问题/Bad Case| S9
    S12 -.->|Prompt 迭代| S8
    S12 -.->|架构瓶颈| S7
```

## 关键依赖说明

1. **核心依赖链**：
   - **用户旅程 (S3)** 是分水岭。在此之前是纯业务分析；在此之后，设计（S4/S6）和技术（S5/S7）可以**并行**工作。
   
2. **提示词设计的依赖**：
   - 必须等待 **功能定义 (S6)** 明确输入输出。
   - 必须等待 **技术选型 (S7)** 确定模型能力（如是否支持 Function Calling）。

3. **评测闭环**：
   - S9 是最重要的迭代点。
   - 如果 Prompt 调优无法解决问题，可能需要回退到 S7 修改架构（如引入 RAG 或更换模型），甚至回退到 S6 修改功能定义（简化任务）。

4. **线上运维闭环**：
   - S12 持续产出 Bad Case 与监控信号，反向驱动 S9/S8/S7 的优化。
