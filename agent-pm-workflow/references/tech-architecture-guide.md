# Agent技术架构设计指南
> 最后更新：2026-02-05（模型与价格为参考示例，请按最新资料更新）

## 一、架构选型决策树

### 1.1 核心模型 (LLM) 选型
> **原则**：在能力满足的前提下，选择成本最低、速度最快的模型。

| 场景 | 推荐模型类型 | 示例模型（参考，需更新） | 考量因素 |
|------|-------------|-----------------|----------|
| **复杂逻辑推理** | SOTA 旗舰模型 | Claude 3.5 Sonnet, GPT-4o, Gemini 2.0 Pro | 逻辑强、工具调用稳定、指令遵循好 |
| **高频简单任务** | 高性价比模型 | Claude 3.5 Haiku, GPT-4o-mini, Gemini 2.0 Flash | 速度快、成本极低 (约旗舰1/10-1/50) |
| **数据隐私敏感** | 开源私有化部署 | Llama 3.3, Qwen 2.5, DeepSeek-V3 | 数据不出域、可微调、支持本地部署 |
| **超长文本处理** | 长上下文模型 | Claude 3.5 (200k), Gemini 2.0 (2M) | 整本书/代码库分析，注意成本线性增长 |
| **多模态任务** | 视觉语言模型 | GPT-4o, Claude 3.5 Sonnet, Gemini 2.0 | 图像理解、文档OCR、视频分析 |
| **代码生成** | 代码专精模型 | Claude 3.5 Sonnet, GPT-4o, DeepSeek-Coder | 代码补全、重构、调试，支持多语言 |

> **成本参考（示例，价格会随时间变化）**：
> - 旗舰模型：输入 $3-15/M tokens，输出 $15-75/M tokens
> - 轻量模型：输入 $0.1-0.5/M tokens，输出 $0.5-1.5/M tokens
> - 开源模型：仅基础设施成本，无API费用

### 1.2 Agent框架选型
| 框架 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **LangChain/LangGraph** | 通用企业级应用 | 生态最丰富、组件标准化、LangGraph支持复杂状态机 | 学习曲线陡峭、抽象层过厚可能导致调试困难 |
| **AutoGen** | 多Agent协作 | 擅长多Agent对话模式、代码执行能力强 | 控制粒度较粗、稳定性需调优 |
| **LlamaIndex** | RAG重度应用 | 数据索引能力极强、检索策略丰富 | Agent控制逻辑相对弱 |
| **CrewAI** | 角色扮演多Agent | 角色定义直观、任务编排简单 | 定制化能力有限 |
| **Semantic Kernel** | 微软生态/.NET | 与Azure深度集成、企业级支持 | 社区相对小、Python支持弱于C# |
| **Dify/Coze** | 低代码快速搭建 | 可视化编排、快速原型验证 | 复杂逻辑受限、厂商锁定风险 |
| **自研/轻量级封装** | 极致可控/特定业务 | 完全掌控Prompt和流程、无冗余 | 开发工作量大、需自己处理边界情况 |

> **选型建议**：MVP阶段优先使用低代码平台验证；生产环境根据团队技术栈选择LangChain(Python)或Semantic Kernel(.NET)。

### 1.3 记忆机制 (Memory) ⭐核心能力
> **记忆是 Agent 区别于普通 LLM 调用的关键能力。**

#### 短期记忆 (Working Memory)
| 策略 | 描述 | 适用场景 |
|------|------|---------|
| **全量保留** | 保留所有历史消息 | 短对话 (<10轮) |
| **滑动窗口** | 只保留最近 N 轮 | 长对话、成本敏感 |
| **摘要压缩** | LLM 定期总结历史 | 超长对话、需保留关键信息 |
| **Token 截断** | 按 Token 数截断 | 简单粗暴但有效 |

#### 长期记忆 (Long-term Memory)
- **向量存储**: 将重要对话/事实 Embedding 后存入向量库，按相关性检索
- **结构化存储**: 用户画像、偏好设置存入关系型数据库
- **知识图谱**: 实体关系存入图数据库 (Neo4j)

#### 记忆设计检查清单
- [ ] 是否需要跨会话记忆？
- [ ] 记忆更新策略是什么？（每轮更新 vs 会话结束更新）
- [ ] 记忆冲突如何处理？（新信息覆盖旧信息？）

## 二、RAG 架构设计 (检索增强生成)

如果Agent需要利用私有数据，必须设计RAG架构。

### 2.1 核心流程
1.  **文档处理**: 清洗 -> 分块 (Chunking) -> Embedding
2.  **检索**: 向量相似度搜索 + 关键词搜索 (混合检索)
3.  **重排 (Rerank)**: 使用 Rerank 模型对粗排结果精选
4.  **生成**: 将Context填入Prompt

### 2.2 关键技术点
-   **分块策略**: 按语义分块还是按字符分块？是否保留重叠 (Overlap)？
-   **向量数据库**: Pinecone (云原生), Milvus (大规模), Chroma (轻量级)
-   **检索优化**: HyDE (假设性文档嵌入), Query Expansion (查询扩展/改写)

## 三、多 Agent 协作模式

当单 Agent 无法胜任复杂任务时，采用多 Agent 协作。

### 3.1 常见协作模式
1.  **顺序流 (Sequential)**: Agent A 输出 -> Agent B 输入 (如: 策划 -> 撰稿 -> 审校)
2.  **层级式 (Hierarchical)**: Boss Agent 拆解任务 -> 分发给 Worker Agents -> 汇总结果
3.  **联合讨论 (Joint Chat)**: 多个专家 Agent (如开发、测试、产品) 在一个群聊中讨论解决问题
4.  **动态路由 (Router)**: 网关 Agent 根据用户意图，将请求路由给最合适的专业 Agent

### 3.2 通信协议
-   **标准化消息格式**: `Source`, `Target`, `Content`, `Thought`, `Action`
-   **共享黑板 (Shared Blackboard)**: 所有 Agent 读写同一份状态/记忆

## 四、工具调用 (Tool Use / Function Calling) ⭐核心能力

> **工具是 Agent 与外部世界交互的桥梁。**

### 4.1 工具定义规范
-   **OpenAPI用于描述**: 标准化接口描述，利于LLM理解。
-   **原子性**: 一个工具只做一件事。
-   **容错性**: 工具内部应处理API失败，返回易读的错误信息给LLM，而不是抛出异常。

### 4.2 工具设计最佳实践
| 原则 | 说明 | 示例 |
|------|------|------|
| **命名清晰** | 动词+名词，见名知意 | `search_products` 而非 `sp` |
| **参数精简** | 必填参数尽量少 | 可选参数用默认值 |
| **返回结构化** | JSON 格式，字段含义明确 | 包含 `success`, `data`, `error` |
| **幂等设计** | 重复调用结果一致 | 查询类工具天然幂等 |

### 4.3 工具调用模式
```
单工具调用: 用户问题 -> LLM选择工具 -> 执行 -> 返回结果
并行调用:   用户问题 -> LLM选择多个工具 -> 并行执行 -> 合并结果
链式调用:   工具A结果 -> 作为工具B输入 -> 最终结果
```

### 4.4 安全沙箱
-   **只读 vs 读写**: 对修改数据的工具（如删除文件、转账）必须增加 `require_human_confirmation` (人工确认)。
-   **代码执行**: 必须在 Docker 容器或受限环境中运行 Python/Shell 代码。

### 4.5 工具设计检查清单
- [ ] 工具描述是否足够清晰让 LLM 理解何时使用？
- [ ] 敏感操作是否有人工确认机制？
- [ ] 工具失败时的降级策略是什么？

## 五、多模态 Agent 架构

当 Agent 需要处理图像、音频、视频等非文本输入时，需要设计多模态架构。

### 5.1 多模态输入处理
| 模态 | 处理方式 | 推荐方案 |
|------|---------|---------|
| **图像** | 直接输入VLM | GPT-4o/Claude 3.5 原生支持图像理解 |
| **PDF/文档** | OCR + 结构化提取 | Azure Document Intelligence, Unstructured.io |
| **音频** | ASR转文本 | Whisper, Azure Speech |
| **视频** | 抽帧 + 图像理解 | 按关键帧采样，结合VLM分析 |

### 5.2 典型应用场景
- **文档问答**: 上传合同/报表图片 -> OCR提取 -> RAG检索 -> 生成回答
- **视觉分析**: 上传产品图 -> 识别缺陷/分类 -> 生成报告
- **会议纪要**: 录音 -> ASR转写 -> 摘要生成 -> 待办提取

## 六、成本优化进阶

### 6.1 Prompt Caching (提示词缓存)
主流厂商已支持缓存长System Prompt，大幅降低重复输入成本：
- **Claude**: Prompt Caching 可降低90%输入成本
- **OpenAI**: Cached Tokens 自动生效
- **适用场景**: System Prompt > 1000 tokens 且高频调用

### 6.2 模型路由 (Model Routing)
```
用户输入 -> 意图分类器(Mini模型) -> 简单问题 -> Mini模型回答
                                  -> 复杂问题 -> Pro模型回答
```
可节省 50-80% 成本。

## 七、反思机制 (Reflection) ⭐核心能力

> **反思是 Agent 自我纠错和持续改进的关键。**

### 7.1 反思类型
| 类型 | 触发时机 | 作用 |
|------|---------|------|
| **即时反思** | 每次输出后 | 检查输出是否符合要求 |
| **错误反思** | 工具调用失败后 | 分析失败原因，调整策略 |
| **周期反思** | 多轮对话后 | 总结进展，调整方向 |
| **终局反思** | 任务完成后 | 评估整体表现，提取经验 |

### 7.2 反思实现模式

**模式1: 自我批评 (Self-Critique)**
```
生成初稿 -> 自我评估 -> 发现问题 -> 修正输出
```

**模式2: 验证循环 (Verify Loop)**
```
生成答案 -> 验证器检查 -> 不通过则重试 (最多N次)
```

**模式3: 多视角审查 (Multi-Perspective)**
```
Agent A 生成 -> Agent B 审查 -> 反馈修正
```

### 7.3 反思 Prompt 示例
```xml
<reflection_instruction>
在给出最终答案前，请先检查：
1. 是否完整回答了用户问题？
2. 是否有事实性错误或幻觉？
3. 是否遗漏了重要信息？
如有问题，请修正后再输出。
</reflection_instruction>
```

### 7.4 反思设计检查清单
- [ ] 是否需要反思机制？（简单任务可能不需要）
- [ ] 反思的成本是否可接受？（会增加 Token 消耗）
- [ ] 反思失败的兜底策略是什么？
